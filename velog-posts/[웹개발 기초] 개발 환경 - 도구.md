<h1 id="백엔드-개발자를-위한-개발-환경--도구-기초-정리">백엔드 개발자를 위한 개발 환경 / 도구 기초 정리</h1>
<p>스프링 기반으로 개발하기 위해서는 언어와 프레임워크 공부 외에, 개발 환경과 주변 도구들에 대한 이해가 필요하다. 이 글에서는 개발자가 반드시 다뤄야 하는 IDE, 버전 관리 도구, DB 도구, CI·Docker 등의 기초 내용을 정리한다.</p>
<hr />
<h2 id="1-ide-intellij-idea-중심으로-이해하기">1. IDE: IntelliJ IDEA 중심으로 이해하기</h2>
<p>백엔드 개발자가 가장 많이 사용하는 IDE는 IntelliJ IDEA이다. 높은 생산성과 코드 분석 기능 덕분에 표준처럼 사용된다.</p>
<h3 id="11-주요-기능">1.1 주요 기능</h3>
<h4 id="코드-네비게이션">코드 네비게이션</h4>
<ul>
<li>Ctrl + 클릭: 정의로 이동</li>
<li>Ctrl + B: 선언으로 이동</li>
<li>Ctrl + Alt + B: 실제 구현체로 이동</li>
</ul>
<p>코드 구조 탐색 속도가 빠르면 생산성이 크게 높아진다.</p>
<h4 id="디버깅">디버깅</h4>
<ul>
<li>Breakpoint 설정</li>
<li>Step Over, Step Into, Resume 기능</li>
<li>변수 상태 확인</li>
</ul>
<p>디버거를 제대로 사용할 줄 아는 것이 유지보수의 핵심이다.</p>
<h4 id="리팩토링-도구">리팩토링 도구</h4>
<ul>
<li>Rename (Shift + F6)</li>
<li>Extract Method (Ctrl + Alt + M)</li>
<li>Move / Change Signature</li>
</ul>
<p>IDE가 제공하는 자동화 리팩토링 기능을 적극 활용하면 안전하게 코드를 개선할 수 있다.</p>
<hr />
<h2 id="2-버전-관리-git-기본과-브랜치-전략">2. 버전 관리: Git 기본과 브랜치 전략</h2>
<p>실무 프로젝트는 모든 변경 사항을 Git으로 관리한다. Git은 단순히 코드를 저장하는 도구가 아니라 협업과 배포 자동화의 기반이 된다.</p>
<h3 id="21-git-기본-흐름">2.1 Git 기본 흐름</h3>
<ol>
<li>로컬 작업</li>
<li>add → commit</li>
<li>push하여 원격 저장소로 업로드</li>
<li>PR 생성 후 팀원의 검토 요청</li>
<li>merge</li>
</ol>
<h3 id="22-대표적인-브랜치-전략">2.2 대표적인 브랜치 전략</h3>
<h4 id="feature-branch">Feature Branch</h4>
<p>단일 기능 개발 시 사용하는 브랜치
예: <code>feature/user-login</code></p>
<h4 id="develop-branch">Develop Branch</h4>
<p>다수 기능을 통합하여 테스트하는 개발용 브랜치</p>
<h4 id="mainmaster-branch">Main(Master) Branch</h4>
<p>배포 가능한 안정된 브랜치
프로덕션의 기준이 된다.</p>
<p>전형적인 전략은 <strong>Git Flow</strong> 또는 <strong>GitHub Flow</strong>를 기반으로 한다.</p>
<h3 id="23-pull-request와-코드-리뷰">2.3 Pull Request와 코드 리뷰</h3>
<ul>
<li>PR은 기능 단위로 작게 나누는 것이 이상적이다.</li>
<li>리뷰 포인트: 변수명, 비즈니스 규칙, 복잡도, 테스트 여부</li>
<li>논리적 근거를 기반으로 서로 토론하는 과정이 중요하다.</li>
</ul>
<hr />
<h2 id="3-로컬-db-도구">3. 로컬 DB 도구</h2>
<p>백엔드 개발에서는 로컬에서 직접 DB를 실행하며 테스트하는 환경이 필요하다. 가장 많이 사용되는 DB는 H2, MySQL, PostgreSQL이다.</p>
<h3 id="31-h2-database">3.1 H2 Database</h3>
<ul>
<li>메모리 기반 또는 파일 기반으로 실행 가능</li>
<li>스프링 부트 개발 환경에서 가장 빠르게 사용</li>
<li>테스트용 DB로 자주 활용</li>
</ul>
<p>예: <code>application.yml</code></p>
<pre><code class="language-yaml">spring:
  datasource:
    url: jdbc:h2:mem:testdb</code></pre>
<h3 id="32-mysql">3.2 MySQL</h3>
<ul>
<li>실무에서 가장 많이 사용하는 관계형 데이터베이스</li>
<li>로컬 설치 또는 Docker로 실행 가능</li>
</ul>
<p>GUI 클라이언트 예:</p>
<ul>
<li>MySQL Workbench</li>
<li>DBeaver</li>
</ul>
<h3 id="33-postgresql">3.3 PostgreSQL</h3>
<ul>
<li>트랜잭션과 복잡한 쿼리에 강함</li>
<li>JPA/Hibernate와 궁합이 매우 좋음</li>
<li>대규모 시스템에서 선호됨</li>
</ul>

<h2 id="4-빌드배포-도구-기초">4. 빌드/배포 도구 기초</h2>
<p>빌드와 배포 자동화를 이해하면 운영 환경까지 연결되는 전체 개발 흐름을 파악할 수 있다.</p>
<h3 id="41-docker-기초-이해">4.1 Docker 기초 이해</h3>
<p>Docker는 애플리케이션을 컨테이너 단위로 패키징하고 실행하기 위한 표준 도구다.</p>
<p>핵심 개념:</p>
<ul>
<li><strong>Dockerfile</strong>: 애플리케이션을 어떻게 패키징할지 정의</li>
<li><strong>Image</strong>: 실행 가능한 패키지</li>
<li><strong>Container</strong>: 실제 실행되는 인스턴스</li>
</ul>
<p>예시 Dockerfile:</p>
<pre><code class="language-dockerfile">FROM openjdk:17
COPY build/libs/app.jar app.jar
CMD [&quot;java&quot;, &quot;-jar&quot;, &quot;app.jar&quot;]</code></pre>
<h3 id="42-ci-continuous-integration">4.2 CI (Continuous Integration)</h3>
<p>CI는 코드 변경이 있을 때 자동으로 빌드, 테스트, 배포를 수행하는 파이프라인이다.</p>
<p>대표 도구:</p>
<ul>
<li>GitHub Actions</li>
<li>GitLab CI</li>
<li>Jenkins</li>
</ul>
<h3 id="43-github-actions-예시">4.3 GitHub Actions 예시</h3>
<p><code>.github/workflows/ci.yml</code></p>
<pre><code class="language-yaml">name: CI Build

on: [push]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-java@v3
        with:
          java-version: '17'
          distribution: 'temurin'
      - name: Build with Gradle
        run: ./gradlew build</code></pre>
<p>코드 변경 → push → 자동 빌드 → 테스트 → 결과 확인의 흐름으로 동작한다.</p>

<h1 id="마무리">마무리</h1>
<p>개발 환경과 도구를 잘 다루는 능력은 백엔드 개발자의 기본 역량이다.
IDE, Git, DB, Docker, CI는 단순 보조 도구가 아니라 개발 전체를 효율적으로 끌어올려주는 핵심 요소들이다.</p>
