<h1 id="java-spring-개발자를-위한-빌드-도구--프로젝트-구조-기초-정리">Java Spring 개발자를 위한 빌드 도구 &amp; 프로젝트 구조 기초 정리</h1>
<p>스프링 기반 애플리케이션을 개발하기 위해서는 빌드 도구와 프로젝트 구조를 이해하는 것이 필수적이다. 이번 글에서는 Maven과 Gradle의 차이, 의존성 관리 방식, 빌드 작동 방식, 그리고 스프링 프로젝트의 표준 디렉터리 구조를 기초적으로 정리한다.</p>
<hr />
<h2 id="1-빌드-도구-개요-maven-vs-gradle">1. 빌드 도구 개요: Maven vs Gradle</h2>
<p>자바 생태계에서 가장 많이 사용되는 빌드 도구는 <strong>Maven</strong>과 <strong>Gradle</strong>이다. 둘 다 프로젝트의 의존성을 관리하고, 빌드와 테스트, 패키징까지 자동화하는 데 사용된다.</p>
<h3 id="11-maven의-특징">1.1 Maven의 특징</h3>
<ul>
<li>XML 기반 설정(<code>pom.xml</code>)</li>
<li>명확한 구조와 설정 방식</li>
<li>관습 중심의 빌드 라이프사이클 제공</li>
<li>기업 환경에서 전통적으로 많이 사용됨</li>
</ul>
<h3 id="12-gradle의-특징">1.2 Gradle의 특징</h3>
<ul>
<li>Groovy 또는 Kotlin DSL 기반 설정(<code>build.gradle</code>)</li>
<li>선언형 + 스크립트형의 유연한 조합</li>
<li>빌드 속도 빠름 (incremental build, 캐시 활용)</li>
<li>현재 스프링 진영에서 가장 많이 사용됨</li>
</ul>
<p>Gradle은 유연성과 속도 면에서 장점이 있어 신규 프로젝트에서는 대부분 Gradle을 선택한다.</p>
<hr />
<h2 id="2-의존성-관리">2. 의존성 관리</h2>
<p>프로젝트 개발에는 라이브러리가 필수적으로 포함된다. 빌드 도구는 이러한 라이브러리(의존성)를 자동으로 다운로드하고 버전을 관리한다.</p>
<h3 id="21-maven-의존성-예시">2.1 Maven 의존성 예시</h3>
<pre><code class="language-xml">&lt;dependency&gt;
    &lt;groupId&gt;org.springframework.boot&lt;/groupId&gt;
    &lt;artifactId&gt;spring-boot-starter-web&lt;/artifactId&gt;
&lt;/dependency&gt;</code></pre>
<h3 id="22-gradle-의존성-예시">2.2 Gradle 의존성 예시</h3>
<pre><code class="language-gradle">implementation 'org.springframework.boot:spring-boot-starter-web'</code></pre>
<h3 id="23-의존성-관리의-핵심">2.3 의존성 관리의 핵심</h3>
<ul>
<li>버전 충돌 해결</li>
<li>트랜지티브 의존성 관리 (하위 의존성 자동 포함)</li>
<li>BOM(Bill of Materials): 스프링 부트는 BOM으로 버전을 자동 관리</li>
</ul>
<p>스프링 부트의 장점은 의존성 버전을 직접 명시할 필요가 거의 없다는 점이다.</p>
<hr />
<h2 id="3-플러그인과-빌드패키징">3. 플러그인과 빌드/패키징</h2>
<h3 id="31-플러그인-역할">3.1 플러그인 역할</h3>
<p>빌드 도구의 기능 확장 모듈이다.</p>
<p>예)</p>
<ul>
<li>스프링 부트 플러그인</li>
<li>테스트 플러그인</li>
<li>JAR/War 패키징 플러그인</li>
</ul>
<h3 id="32-jar-패키징">3.2 JAR 패키징</h3>
<p>애플리케이션 코드를 하나의 실행 파일로 묶는 과정이다.</p>
<p>Maven:</p>
<pre><code class="language-bash">mvn package</code></pre>
<p>Gradle:</p>
<pre><code class="language-bash">./gradlew build</code></pre>
<p>출력물은 보통 다음 경로에 생성된다:</p>
<pre><code>build/libs/프로젝트명-version.jar</code></pre><h3 id="33-spring-boot-jar">3.3 Spring Boot Jar</h3>
<p>Spring Boot는 실행 가능한 JAR를 만들기 때문에 다음 명령으로 바로 실행할 수 있다.</p>
<pre><code class="language-bash">java -jar build/libs/app.jar</code></pre>
<hr />
<h2 id="4-spring-initializr로-프로젝트-생성해보기">4. Spring Initializr로 프로젝트 생성해보기</h2>
<p>실습을 통해 빌드 도구와 프로젝트 구조를 확인하는 것이 가장 이해가 빠르다.</p>
<ol>
<li><a href="https://start.spring.io">https://start.spring.io</a> 에 접속</li>
<li>Gradle/Groovy 또는 Gradle/Kotlin 선택</li>
<li>Spring Boot 버전 선택</li>
<li>Dependencies에서 Web, JPA, Validation 등 선택</li>
<li>프로젝트 생성 후 열어보기</li>
</ol>
<p>생성된 프로젝트는 스프링에서 권장하는 표준 구조가 그대로 적용되어 있다.</p>
<hr />
<h2 id="5-프로젝트-표준-디렉터리-구조">5. 프로젝트 표준 디렉터리 구조</h2>
<p>스프링은 Maven/Gradle 공통 관습을 적용해 다음 구조를 기본으로 사용한다.</p>
<pre><code>src/
 └── main/
      ├── java/
      ├── resources/
 └── test/</code></pre><h3 id="51-srcmainjava">5.1 src/main/java</h3>
<p>애플리케이션의 실제 Java 코드가 위치한다.
컨트롤러, 서비스, 리포지토리, 도메인 등 모든 계층 코드가 여기에 들어간다.</p>
<h3 id="52-srcmainresources">5.2 src/main/resources</h3>
<p>설정 파일 및 정적 리소스가 포함된다.</p>
<ul>
<li><code>application.yml</code> 또는 <code>application.properties</code></li>
<li>SQL 스키마 파일</li>
<li>정적 HTML, 템플릿</li>
</ul>
<h3 id="53-srctestjava">5.3 src/test/java</h3>
<p>모든 테스트 코드가 들어간다.
스프링에서는 JUnit 기반 테스트를 사용한다.</p>
<hr />
<h2 id="6-패키지-설계-기본-구조">6. 패키지 설계 기본 구조</h2>
<p>스프링 애플리케이션에서 가장 흔히 사용하는 패키지 구조는 도메인 기반 레이어드 아키텍처다.</p>
<pre><code>com.example.project
 ├── controller
 ├── service
 ├── repository
 ├── domain (또는 entity)
 ├── dto
 ├── config</code></pre><h3 id="61-controller">6.1 Controller</h3>
<p>HTTP 요청을 받고 응답을 반환하는 레이어
주 역할: 외부 요청 → 서비스 호출 → 응답 DTO 변환</p>
<h3 id="62-service">6.2 Service</h3>
<p>비즈니스 로직을 처리하는 핵심 계층
도메인 규칙, 트랜잭션 관리가 위치한다.</p>
<h3 id="63-repository">6.3 Repository</h3>
<p>데이터베이스 접근 계층
JPA에서는 <code>JpaRepository</code> 인터페이스를 상속해 구현한다.</p>
<h3 id="64-domainentity">6.4 domain/entity</h3>
<p>데이터베이스와 매핑되는 도메인 모델</p>
<p>예)</p>
<pre><code class="language-java">@Entity
public class User {
    @Id
    private Long id;
}</code></pre>
<h3 id="65-dto">6.5 dto</h3>
<p>컨트롤러 입출력용 객체
도메인(Entity)와 직접 노출을 분리하는 역할</p>
<h3 id="66-config">6.6 config</h3>
<p>스프링 설정 클래스
Swagger, CORS, Security 설정 등이 여기에 위치한다.</p>
<hr />
<h1 id="마무리">마무리</h1>
<p>스프링 개발의 첫 단계는 문법보다 “프로젝트의 구조와 빌드 시스템을 이해하는 것”이다.
특히 Gradle 기반의 의존성 관리와 패키징 개념을 이해하면, 이후 Spring MVC, JPA, Security 등으로 확장할 때 큰 도움을 받을 수 있다.</p>
